# 跨介质抓取项目后续推进规划

更新日期：2026-05-27

## 1. 当前判断

项目已经完成了物理属性 raw 表的整理，并已生成正式物理属性表：

```text
data/processed/object_physical_attributes.csv
```

这一步意味着对象级物理属性已经具备进入数据管线的条件。接下来的关键不是立刻把物理属性输入策略头，而是先证明：

1. 新物理标签是稳定、可复现、可校验的；
2. 视觉/触觉 content branch 能否学习这些负载相关物理属性；
3. 物理属性辅助监督是否能改善 Interface 阶段的控制表现；
4. 真实或预测物理属性是否有必要进一步进入 policy。

因此，推荐采用：

```text
先证据 -> 后接入 -> 再消融
```

的推进节奏。

## 2. 第一阶段：数据管线收口

### 2.1 接入正式物理属性表

将以下文件接入 Dataset：

```text
data/processed/object_physical_attributes.csv
```

每个 sample/window 应该通过 `object_id` join 到对象级物理属性，至少输出：

```text
dry_mass_g
is_open_container
full_water_mass_g
water_capacity_g
capacity_valid_mask
capacity_ratio
load_transition_class
shape_profile_ctrl
material_safety_class
surface_texture
```

### 2.2 连续属性归一化

连续物理属性建议使用 train split 的统计量归一化：

```text
dry_mass_g
water_capacity_g
capacity_ratio
```

需要保存归一化统计量，建议路径：

```text
data/processed/stats/physical_attribute_norm_stats.json
```

注意：`water_capacity_g` 只对开口容器有意义，因此对应 loss 和 metric 必须使用：

```text
capacity_valid_mask
```

非开口对象不应该被强迫回归容量。

### 2.3 保留旧属性标签

旧的 `fragility / geometry / surface` 暂时不要删除。建议让 Dataset 同时支持：

```text
old semantic attributes
new physical attributes
```

这样后续可以做：

```text
old_attr_vs_physical_attr
```

对照实验。

## 3. 第二阶段：物理属性头 Sanity Check

### 3.1 第一轮优先预测目标

物理属性头第一轮建议只预测最核心的负载相关属性：

```text
dry_mass_g
capacity_ratio
is_open_container
```

对应任务类型：

```text
dry_mass_g: regression
capacity_ratio: regression
is_open_container: binary classification
```

`water_capacity_g` 可以同时预测，但必须使用 `capacity_valid_mask`。

### 3.2 第二轮可选预测目标

如果第一轮 sanity check 正常，再加入：

```text
shape_profile_ctrl
load_transition_class
```

其中 `shape_profile_ctrl` 是从旧 `geometry` 映射来的控制相关粗形状标签；`load_transition_class` 是由质量、容量和容量比派生的负载变化风险等级。

### 3.3 不要立即进入 Policy

第一版物理属性头只作为：

```text
auxiliary supervision
mechanism diagnostic
representation probing
```

暂时不要把真实属性或预测属性输入 `delta head` 或 `gate head`。

这样可以避免两个风险：

1. 审稿人质疑测试时是否需要已知重量/容量；
2. 属性头效果不稳定时污染主控制策略。

## 4. 第三阶段：Window / Sample / Object 三级评估

物理属性是对象级稳定属性，不是严格的窗口级标签。因此不能只看 window-level 指标。

建议同时报告：

```text
window-level metrics
sample-level aggregation metrics
object-level aggregation metrics
```

### 4.1 Window-level

用于检查模型在单个窗口上的即时可预测性。

问题是：单窗口可能受视觉角度、触觉接触片段和 Interface 扰动影响较大。

### 4.2 Sample-level

将同一个 sample 内的多个窗口预测进行聚合：

```text
mean pooling
max pooling
attention pooling
```

优先实现 mean pooling，作为第一版稳定基线。

### 4.3 Object-level

将同一对象的多个 sample 继续聚合，得到 object-level 预测。

这更符合物理属性的定义，也更适合向专家解释：

```text
物理属性是对象级属性，窗口级预测只是诊断视角。
```

## 5. 第四阶段：Split 检查与重构

现在物理属性分布已经明确，必须重新检查 train/val/test 是否在以下维度上严重偏斜：

```text
dry_mass_g
capacity_ratio
is_open_container
load_transition_class
shape_profile_ctrl
```

尤其需要避免：

```text
高容量对象全部在 test
重物体全部在 test
non_open_container 全部集中在某个 split
constricted_opening 形状集中在某个 split
```

推荐构建：

```text
mass/capacity-balanced object-level unseen split
```

主实验仍然保持 object-level unseen，但 split 需要尽量平衡物理属性分布。

如果属性头在 unseen split 下表现很弱，可以增加一个诊断用 split：

```text
object-overlapping split
```

但该 split 只用于属性头可学性诊断，不作为主控制泛化结论。

## 6. 第五阶段：属性辅助对控制的影响

当物理属性头 sanity check 通过后，再比较：

```text
ref_anchor_no_attr_aux
ref_anchor_physical_attr_aux
```

其中 `physical_attr_aux` 表示物理属性只作为辅助监督，不进入 policy。

重点观察 Interface 阶段控制指标：

```text
control_interface_mae
control_interface_hit_rate_200
control_interface_bias
delta_minus_target.abs_mean
force_interface_delta_mean
non_interface_delta_magnitude
```

如果辅助属性头能改善 Interface 阶段表现，论文叙事会更稳：

```text
物理属性辅助 content branch 编码负载相关因素，
主控制仍由 reference anchor + medium-gated residual 完成。
```

## 7. 第六阶段：Metadata 消融

metadata 消融应放在后面，不建议作为第一轮实验。

建议做三组：

```text
no_metadata
oracle_metadata
predicted_metadata
```

含义如下：

| 实验 | 含义 | 目的 |
|---|---|---|
| `no_metadata` | 不向 policy 输入物理属性 | 当前主线 |
| `oracle_metadata` | 向 policy 输入真实物理属性 | 判断物理属性对控制的上限价值 |
| `predicted_metadata` | 向 policy 输入属性头预测值 | 判断预测属性是否能实际服务控制 |

解释方式：

```text
oracle_metadata 提升明显，predicted_metadata 不提升：
说明物理属性有价值，但属性头预测质量是瓶颈。

oracle_metadata 也不提升：
说明 reference_force 可能已经吸收了主要对象负载差异。

predicted_metadata 提升：
说明属性头不仅可解释，还可能对控制有直接帮助。
```

## 8. 推荐实验顺序

建议按照以下顺序推进：

```text
1. physical_attr_dataset_sanity
2. physical_attr_stage1_window_eval
3. physical_attr_stage1_sample_object_eval
4. balanced_split_check
5. physical_attr_stage2_aux
6. ref_anchor_no_attr_aux
7. ref_anchor_physical_attr_aux
8. oracle_metadata_ablation
9. predicted_metadata_ablation
```

其中前 5 项是近期优先级最高的工作。

## 10. 总体建议

当前项目最适合继续走以下路线：

```text
reference anchor + medium-gated residual policy 作为主控制方法
physical attributes 作为辅助监督和机制解释
metadata policy input 作为后续消融
```

这样推进的好处是：

1. 主线清晰，不会让物理属性喧宾夺主；
2. 属性头失败时不会破坏主控制结论；
3. 属性头成功时可以增强论文的机制解释；
4. metadata 消融能回答物理属性是否真的有控制上限价值；
5. 整体叙事更容易向专家解释。

一句话总结：

```text
下一阶段应优先证明物理属性标签可靠、可学、能解释控制差异；
在证据充分之后，再考虑让真实或预测物理属性进入 policy。
```
