# 策略训练诊断现状

更新日期：2026-05-05

## 1. 当前项目阶段

项目当前已经不是“Stage1-Stage4 能不能跑通”的问题，而是“策略力输出怎样才能稳定学好”的问题。

当前整体状态可以概括为：

- 离线工程闭环已完成，数据预处理、训练、评估、诊断脚本都可用。
- 感知侧已经有效，尤其 `Interface` 识别具备明显能力。
- 当前主瓶颈是策略控制力输出，不是感知主链本身。

## 2. 已确认的证据链

### 2.1 策略问题适合拆成 `base/reference` 和 `delta`

当前策略输出更像：

```text
force_pred = base_force + interface_gate * interface_delta
```

其中：

- `base/reference` 负责稳定阶段基准力
- `delta` 负责 `Interface` 段相对基准力的修正量

这比把“最终控制力”当成一个单头回归问题更符合实验观察。

### 2.2 `delta` 本身是能学的

OBJ012 tiny 过拟合实验已经说明：

- `tiny_delta`：几乎学不起来，`delta` 长期贴近 0
- `tiny_oracle_delta`：只要 `base` 换成 `oracle/reference`，`delta` 立刻能学起来
- `tiny_oracle_delta_norm`：再加 `delta` 尺度归一化后更稳

结论：

- `delta` 分支本身不是“不会学”
- 它依赖一个稳定锚点

### 2.3 `base` 在 tiny 上能学，但对尺度很敏感

OBJ012 tiny 结果说明：

- `tiny_base`：能往正确方向走，但不容易贴住 `reference_force`
- `tiny_base_norm`：给 base loss 做归一化后，`base/reference` 子问题可以学顺

结论：

- `base` 不是完全学不会
- 但它对目标量级、学习率和 loss 工作区间很敏感

### 2.4 `learned base frozen + delta_norm` 和 `light joint` 都说明 joint 不是唯一主问题

OBJ012 tiny 上已经完成：

- `learned base frozen + delta_norm`
- `light joint fine-tune`

结论是：

- `learned base` 可以托住 `delta`
- `light joint` 可以把 train 上的 `base` 和 `delta` 接顺
- 但 val 泛化短板主要仍然在 `base`

也就是说，之前的主矛盾不是“`delta` 死活学不会”，而是“`base` 不稳会把后面一起拖坏”。

## 3. `reference_force` 分组统计带来的关键转折

`fold1` 上的 `reference_force` 统计已经说明：

- `reference_force` 不是稳定的对象级常数
- 它更像强烈依赖当前 episode 的样本级量

已观察到的结构性现象包括：

- train `21` 个分组里，`14` 个分组 `reference_force_std >= 100`，`8` 个分组 `>= 200`
- val `7` 个分组里，`5` 个分组 `reference_force_std >= 100`
- train 的 `6` 个对象里，全部对象级 `reference_force_std >= 200`

同对象、同条件组内也会明显漂移，例如：

- `OBJ005 / clear + normal + normal`：约 `-806.8 -> -449.2`
- `OBJ010 / clear + normal + inverse`：约 `-395.7 -> -134.3`

这意味着：

**让模型纯猜一个 absolute `base`，不仅优化难，而且目标本身就不够稳。**

## 4. `fold1_base_norm` 的结论：pure learned absolute base 不适合作为当前主线

对应配置：

`configs/stages/stage36b_policy_fold1_base_norm.yaml`

结果表明：

- `val_control_reference_mae ≈ 520.5`
- `val_control_interface_mae ≈ 519.3`
- `val_control_interface_hit_rate_200 ≈ 0.0577`

更关键的是，诊断里：

- `reference_force_mean ≈ -539.2`
- `force_base_mean ≈ -4.8`
- `base_minus_reference_mean ≈ +534.4`

也就是：

**`base` 基本还停在 0 附近，没有真正学到 `reference_force` 的量级。**

这条 run 的价值不在于“效果好”，而在于它很明确地告诉我们：

- pure learned absolute base 仍然值得保留为 baseline
- 但它不应该继续被当成最有希望的主线

## 5. 最新关键结果：`reference_force` 直锚在 fold1 上被强力验证

对应配置：

`configs/stages/stage36b_policy_fold1_reference_anchor_delta_norm.yaml`

这条配置的核心是：

- `base_source: reference_force`
- 冻结 `policy_head.base_*`
- 只训练 `delta`
- 使用 `delta_normalization_scale: 150.0`

和 `fold1_base_norm` 相比，控制指标出现数量级改善：

- `val_control_reference_mae: 520.5 -> 56.7`
- `val_control_interface_mae: 519.3 -> 68.9`
- `val_control_interface_hit_rate_200: 0.0577 -> 0.9375`
- `val_control_interface_bias: +513.9 -> -16.9`

最关键的分支诊断是：

- `base_minus_reference_mean = 0.0`

这说明：

**一旦把 `base` 直接交给 measured `reference_force`，之前那个最大的泛化坑被结构性绕开了。**

### 5.1 这条结果该怎么解读

这条 run 的 `standard_metrics` 依然会看起来比较差，例如 `overall_mae ≈ 559`。  
这不代表方案失败，因为这组指标是直接按 raw `expert_force` 在算。

当前更该看的，是：

- `control_reference_mae`
- `control_interface_mae`
- `control_interface_hit_rate_200`
- `base_minus_reference`
- `delta_minus_target`

也就是说：

**对直锚方案，主评价口径应该是 control 指标和分支级诊断，而不是 raw expert 的 headline MAE。**

### 5.2 这条结果也暴露了剩余短板

当前 `base` 已经不再是主问题，但 `delta` 还没有完全学透。

在 `interface_expert` 上可见：

- `force_interface_delta_mean ≈ 2.4`
- `delta_target_mean ≈ 18.7`
- `delta_minus_target.abs_mean ≈ 68.9`

这说明现在更像是：

- 锚点已经对了
- `delta` 方向也开始对了
- 但幅值还偏保守，没有完全长起来

## 6. 当前最可信的总体判断

截至 2026-04-28，最可信的判断是：

1. `delta` 是能学的
2. pure learned absolute `base` 在正式 `fold1` 上不适合作为当前主线
3. measured `reference_force` 作为锚点是当前最有效的结构性改进
4. 当前主瓶颈已经从“`base` 学不学得会”收缩成：

**“在 reference 锚点已经给定的前提下，如何把 `delta` 学得更充分、更稳。”**

## 7. 当前推荐的优化方向

### 7.1 短期：保留 pure learned base 路线作为 baseline

继续保留这条三阶段路线：

1. `base-only normalized`
2. `learned base frozen + delta_norm`
3. `light joint fine-tune`

但它现在的定位是：

- 正式 baseline
- 不是默认主线

### 7.2 主线切换到 `reference_force` 直锚

当前更应该投入的方向是：

```text
force_pred = reference_force + gate * delta
```

下一步再逐渐扩展成：

```text
force_pred = reference_force + base_correction + gate * delta
```

这样做更符合当前证据，因为：

- `reference_force` 本身就是 episode 级锚点
- 不需要再逼模型盲猜 absolute base
- 模型只需要学“在当前锚点上补多少”

### 7.3 stronger 版本应该重点加强 `delta`，不是再回头折腾 `base`

既然当前 `base` 已经通过结构被稳定住，那么 stronger 配置应该主要做这些事：

- 提高 `residual_interface_weight`
- 适当降低 `residual_stable_zero_weight`
- 适当降低 `residual_non_interface_weight`
- 稍微提高 interface 采样强度
- 延长训练轮数

这比重新把精力放回 pure learned base 更值得。

### 7.4 数据治理优先继续看 reference 质量和可疑样本

当前仍建议优先检查：

- `reference_force_window_count`
- `reference_force_statistic`
- reference window 是否真的落在稳定段
- 极端样本是否带偏 reference

优先审查可先看：

- val：`S0160`, `S0163`, `S0078`, `S0080`, `S0076`
- train：`S0016`, `S0037`, `S0039`, `S0044`

## 8. 当前建议的下一步

当前最推荐的下一步是：

1. 保留 `stage36b_policy_fold1_base_norm` 作为正式 baseline
2. 继续推进 `reference_anchor` 主线
3. 跑更强一些的 `fold1 reference_anchor_delta_stronger`
4. 继续固定看分支级指标：
   - `base_minus_reference`
   - `delta_minus_target`
   - `force_base`
   - `force_interface_delta`
   - `predicted_gate`
   - `control_interface_mae`

## 9. 一句话总结

到 2026-04-28 这个阶段，最重要的结论已经不是“再试一个 full joint 会不会更好”，而是：

**measured `reference_force` 直锚在 fold1 上已经被强力验证；当前最值得推进的主线，是让 `reference_force` 负责锚点，再重点把 `delta` 学得更充分。**

## 10. 2026-05-05 阶段更新

从 2026-04-29 到 2026-05-05，主线继续围绕：

```text
force_pred = reference_force + gate * delta
```

展开，但已经完成了几轮更细的排查。当前判断比 2026-04-28 更收敛：`reference_force` 直锚仍然是主线，新的关键问题不再是“是否要引入 context 或 oracle gate”，而是“怎样保留 medium gate 的有效提前/滞后激活，同时抑制 stable 外溢”。

### 10.1 `balanced` 成为当前稳定基线

当前稳定基线配置为：

`configs/stages/stage36b_policy_fold1_reference_anchor_delta_balanced.yaml`

它相对 `norm` 和 `stronger` 的定位是：

- `norm`：太保守，`delta` 补得不够
- `stronger`：推进了一步，但 stable 外溢开始变明显
- `balanced`：在两者之间取得目前最稳的折中

`balanced` 的 val 关键指标：

- `control_interface_mae = 68.3265`
- `control_interface_hit_rate_200 = 0.9399`
- `control_interface_bias = -9.44`

从 `stronger best.pt` 初始化再跑 `balanced_from_stronger`，结果基本回到同一个最优点，没有形成新的更好 basin。因此，继续在 `stronger -> balanced` 这条线上微调意义不大。

### 10.2 界面增强 `V1/V1b` 的结论

已实现并测试：

- `stage36b_policy_fold1_reference_anchor_delta_context_v1.yaml`
- `stage36b_policy_fold1_reference_anchor_delta_context_v1b.yaml`

V1 的设计是给 residual/delta 分支增加界面前 stable context embedding，不改 target 语义，不改 `reference_force` 直锚，不让 context 影响 gate。

结果：

- `V1 control_interface_mae = 68.3318`
- `V1b control_interface_mae = 68.3399`
- 两者均未超过 `balanced`

关键诊断：

- V1 中 context 支路几乎没有打开，`residual_context_scale` 接近 0
- V1b 通过非零初始化和更高 LR 让 context 支路参与了计算
- 但 V1b 主要表现为“整体更激进”，interface 推力和 stable 外溢同时增加

结论：

**当前 simple additive context 路线不值得继续小幅调参。**

这不等于“界面前 context 没价值”，而是说明 raw context 直接加到 residual 上，不能精准解决当前问题。若未来继续做 context，更合理的方向是 `current - stable_context` 的差分式 context，而不是简单 additive context。

### 10.3 reference quality 诊断已经补齐

新增脚本：

`scripts/diagnose_reference_quality.py`

当前在 `stage36c_reference_anchor_oracle_gate_delta.yaml` 的 val 上运行结果：

- `sample_count = 30`
- `reference_sample_count = 30`
- `bad_reference_count = 0`

输出包括：

- `reference_quality_samples.csv`
- `reference_quality_group_summary.csv`
- `reference_quality_object_summary.csv`
- `bad_reference_cases.json`
- `reference_force_distribution.png`

结论：

**至少在当前 val 阈值下，`F_ref` 没有明显脏样本；下一步瓶颈不优先归因于 reference 质量。**

### 10.4 soft oracle gate 的结论

已新增 soft gate target 构造：

`scripts/build_soft_gate_targets.py`

当前参数：

- `soft_gate_pre_sec = 0.3`
- `soft_gate_post_sec = 0.5`
- `soft_gate_ramp = cosine`

val 导出统计：

- `row_count = 2737`
- `nonzero_count = 472`
- `full_gate_count = 380`

已测试配置：

- `stage36c_reference_anchor_oracle_gate_delta.yaml`
- `stage36c_b_reference_anchor_oracle_gate_gated_delta.yaml`

结果对比：

| run | control_interface_mae | hit_rate_200 | bias | stable_leakage |
|---|---:|---:|---:|---:|
| `balanced` | `68.3265` | `0.9399` | `-9.44` | 未显式记录 |
| `36c soft oracle` | `68.4532` | `0.9327` | `-5.62` | `0.0` |
| `36c_b soft gated-loss` | `68.4539` | `0.9327` | `-5.61` | `0.0` |

结论：

**soft oracle gate 可以把 stable 外溢压到 0，但没有提升 interface 主控制指标。**

`36c_b` 把 loss 改成更强调 `gate * delta -> delta_target` 后，几乎没有改变结果，说明问题不只是 raw delta 和 gated residual 监督之间的轻微不匹配。

### 10.5 hard oracle gate 的结论

已测试配置：

`stage36c_c_reference_anchor_hard_oracle_gate_delta.yaml`

这版只改一件事：

```yaml
gate_source: hard_phase
```

结果：

- `control_interface_mae = 68.3790`
- `control_interface_hit_rate_200 = 0.9327`
- `control_interface_bias = -4.98`
- `stable_leakage_mean = 0.0`
- `stable_leakage_p95 = 0.0`

和 soft oracle 相比，hard oracle 稍好；但仍然没有超过 `balanced`。

结论：

**oracle gate 不是下一步突破口。**

它能让 stable 段更干净，但会损失一部分 interface 主指标。这说明当前 `medium_prob` gate 的提前/滞后/软激活虽然带来一些 stable 外溢，但对 interface 控制是有帮助的。

### 10.6 诊断口径小修正

由于 dataset 现在总会带 `soft_gate_targets`，`diagnose_policy_control.py` 里原来的 `oracle_gate_metrics` 在 hard gate stage 下会误用 soft target 作为 oracle gate。

已修正为：

- 当 `model.policy.gate_source == soft_target` 时，诊断 oracle 使用 `soft_gate_targets`
- 否则默认使用 hard phase gate

注意：这个问题只影响诊断脚本里 `oracle_gate_metrics` 的辅助口径，不影响训练结果，也不影响 `best_metrics.json` 里的 `control_*` 主指标。

## 11. 当前最新判断

截至 2026-05-05，最可信的判断更新为：

1. `reference_force` 直锚仍是主线，`balanced` 是当前稳定基线。
2. 继续在 `balanced/stronger` 局部微调，边际收益已经很低。
3. simple additive context 没有带来净收益，不应继续小调参。
4. soft/hard oracle gate 都能压外溢，但都没有超过 `balanced`。
5. 当前真正值得推进的是：保留 `medium_prob` gate 的有效动态，同时加稳定器压 stable 外溢。

当前建议主线从：

```text
reference_anchor_delta_balanced
-> soft/hard oracle gate
```

更新为：

```text
reference_anchor_delta_balanced
-> medium_prob gate + PhaseAwareStabilizer
-> stable leakage loss / metrics
-> learned gate only if stabilizer route proves useful
```

也就是说，下一步不建议直接进入 learned gate。因为 oracle gate 已经说明“更干净的 gate”不自动带来更好的 interface 控制。

## 12. 当前下一步

当前已新增但尚待正式跑完的配置：

`configs/stages/stage36d_reference_anchor_medium_gate_stabilized_delta.yaml`

这版保留：

```yaml

```

并新增：

```yaml

  enabled: true
  on_threshold: 0.20
  off_threshold: 0.08
  min_on_windows: 1
```

目标是：

- 保留 medium gate 在 interface 附近的提前/滞后激活
- 抑制 stable 段小 gate 带来的 residual 外溢
- 尽量不牺牲 `control_interface_mae` 和 `hit_rate_200`

## 13. 更新后一句话总结

到 2026-05-05，主线已经进一步收敛为：

**`reference_force` 负责 episode 锚点，`delta` 负责界面修正；下一步不再追求更“干净”的 oracle gate，而是保留 `medium_prob` gate 的有效动态，并用 phase-aware stabilizer 控制 stable 外溢。**
