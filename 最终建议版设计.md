## 1. 文档目的

本文档用于把工程师提供的设计建议，整理成一个更适合当前项目继续推进的“最终建议版”。

目标不是简单复述原建议，而是回答三个问题：

1. 这份建议整体上是否合理
2. 哪些部分应该直接采纳为主线
3. 哪些部分适合作为后续实验或工程增强，而不是立刻成为主方法

本文档面向：

- 负责后续模型迭代的算法工程师
- 负责数据与标签管线的工程师
- 负责论文主线组织的研究人员

## 2. 总体判断

总体上，我认为这份建议是**合理的，而且质量很高**。

它最有价值的地方在于：

1. 没有再回到“从零预测绝对抓取力”的旧路线
2. 没有把完整专家切换系统过早抬成主方法
3. 把当前最有证据支持的主线，明确收束为：

```text
F_pred(t) = F_ref(e) + β_if(t) · ΔFθ(t)
```

4. 同时把专家切换方案里真正有价值的工程机制，以“软门控稳定化”和“安全层”的形式吸收进来

从方法论上说，这份建议做对了两件事：

- 在论文主线层面保持方法贡献清晰
- 在工程部署层面保留了未来扩展空间

因此，**这份建议值得采纳，并且适合作为当前项目后续推进的统一版本。**

## 3. 我建议直接采纳的核心结论

## 3.1 不把完整专家切换系统作为当前论文主方案

这是我最认同的一点。

原因不是专家切换不合理，而是：

- 它会显著增加系统复杂度
- 会把贡献重心从“跨介质残差力适应”转移到“状态机 + 混合专家调度”
- 以当前数据规模，完整多专家系统会更容易把数据切碎

因此当前阶段更合适的做法是：

- **论文主方法**：`reference anchor + soft-gated residual delta`
- **工程增强层**：吸收专家切换思想中的平滑切换、滞回、驻留时间、安全过滤

## 3.2 以 `reference_force` 作为 episode-level 锚点

这一点与我们当前实验结论高度一致。

关键理解应为：

- `reference_force` 不是对象级常数
- 它是当前 episode 在界面前稳定抓持阶段的样本级控制锚点

因此当前主方法不应继续尝试学习一个不稳定的 absolute base，而应直接使用：

```text
F_ref(e)
```

作为因果可获得的控制基线。

## 3.3 主模型只学习 residual delta，不重学完整力值

这条建议非常重要，也非常合理。

主问题应从：

```text
视觉 + 触觉 -> 绝对抓取力
```

改写为：

```text
视觉 + 触觉 + 介质信念 + 参考锚点 -> 界面相关修正量
```

这会让任务从一个“绝对量大回归问题”转成一个“结构化相对修正问题”，更符合：

- 当前数据规模
- 当前实验观察
- 当前已知瓶颈

## 3.4 Gate 必须是软门控，不应只用硬标签

这条建议同样应采纳。

不建议使用：

```text
gate = 1 if interface else 0
```

建议使用：

```text
β_if(t) = SmoothGate(p_interface, approach_score, history, phase_context)
```

原因是跨界面控制本身就是一个短时过程，不是一个单点事件。

软门控的作用不是“把分类概率变好看”，而是：

- 界面前提前介入一点
- 界面中充分释放 residual
- 界面后平滑退出
- 减少 stable 段误激活和输出突变

## 3.5 吸收专家切换方案中的稳定化机制，但不引入完整专家切换架构

建议采纳以下思想，但以**后处理稳定器**而不是“完整专家系统”的方式进入主线：

- smoothing
- hysteresis
- minimum dwell time
- soft ramp
- safety filtering

这部分可以抽象成：

```text
β_raw(t) -> PhaseAwareStabilizer -> β_if(t)
```

它的角色是：

- 稳定 gate
- 抑制 residual 外溢
- 降低界面前后切换抖动

## 4. 我建议谨慎采纳的部分

## 4.1 `base_correction` 只作为后续扩展，不作为第一版主方法

工程师建议中已经写得比较克制，我同意这个判断。

如果后续发现：

- `F_ref` 在某些长时段存在系统偏差
- 或 stable_air 阶段有一致性偏差

可以扩展为：

```text
F_pred = F_ref + ε_stable + β_if · ΔF
```

但这里的 `ε_stable` 必须是：

- 小幅
- 强约束
- 明确服务稳态补偿

不能重新滑回“learned absolute base”的老路。

因此：

- **第一版不作为主方法**
- **可作为轻量扩展或 B-lite 对照**

## 4.2 触觉 high / low 解耦应保留为方法主张，但实现节奏要稳

工程师建议把：

- high tactile -> 接触内容 / 动态扰动
- low tactile -> 介质信念 / 低频趋势

写成论文贡献点，这在概念上是成立的。

我同意保留这条方法主张，但实现优先级上建议注意：

1. 如果现有仓库已经具备相近结构，则直接沿用并增强
2. 如果现有实现还不够明确，不建议为了“形式完整”一次性大改所有触觉编码器

更稳的顺序应该是：

- 先把 `reference anchor + gate + delta + leakage control` 主线跑顺
- 再进一步强化 high/low tactile 的显式解耦

也就是说：

- **论文上可以保留为核心设计思想**
- **工程上要注意实施节奏**

## 4.3 `B-lite` 双头结构可以做，但只作为轻量对照

我认可工程师提出的：

```text
F_pred = F_ref + ε_stable + β_if · ΔF_interface
```

这一轻量双头版本。

但它更适合作为：

- 轻量 expert-thinking ablation

而不是当前主结果。

只有在它显著优于主线时，才值得进一步上升为主方案讨论对象。

## 5. 我建议暂不作为当前主线的部分

## 5.1 完整 Full Expert Switching

当前不建议把以下结构作为主实验：

- stable expert
- interface expert
- phase manager
- expert fusion
- safety logic

原因与工程师建议一致，而且我认为判断正确：

1. 数据会被切碎
2. 训练和消融变量会明显增多
3. 论文主线会被系统设计复杂度稀释
4. 当前已经有更直接有效的 `reference_anchor` 证据链

这条路线更适合作为：

- deployment extension
- future work
- supplementary design

## 5.2 重新回到纯 absolute force regression

不建议再把以下形式作为主线回退：

```text
F_pred = Policy(obs)
```

它在当前问题设定下不再是结构上最合适的定义。

## 6. 最终建议采用的方法定义

我建议将项目最终主方法明确定义为：

## 6.1 主控制公式

```text
F_pred(t) = F_ref(e) + β_if(t) · ΔFθ(t)
```

其中：

- `F_ref(e)`：界面前稳定抓持阶段得到的 episode-level 参考力
- `ΔFθ(t)`：当前时刻的界面相关残差修正量
- `β_if(t)`：由介质信念和阶段上下文决定的软门控权重

## 6.2 方法名称建议

建议名称可统一为以下其一：

1. `Cross-Medium Reference-Anchored Residual Force Adaptation`
2. `CMG-RAF: Cross-Medium Grasping with Reference-Anchored Force Adaptation`

## 6.3 方法角色定义

该方法应被描述为：

**一个以 episode-level reference force 为控制锚点，通过介质信念门控 residual force correction，实现跨介质抓取力自适应的视触控制系统。**

而不是：

- 一个端到端绝对力回归器
- 一个完整专家切换混合控制系统

## 7. 建议保留的必要设计细节

## 7.1 `reference_force` 的构造原则

建议定义：

```text
stable_pre_interface_window(e) = [t_grasp_stable, t_if_enter - margin]
```

并从该窗口中构造：

```text
F_ref(e) = RobustAggregate(F[t])
```

推荐聚合方式：

- median
- trimmed mean
- EMA over stable window

不建议使用界面后信息来构造 `F_ref`，避免目标泄漏。

## 7.2 residual target

建议定义：

```text
ΔF_target(t) = F_target(t) - F_ref(e)
```

这样 residual head 学的是“相对修正量”，而不是绝对值。

## 7.3 soft gate target

建议基于以下时间点构造：

- `t_if_enter`
- `t_if_exit`
- `Δpre`
- `Δpost`

软门控目标形式建议为：

```text
β_target(t) = 0                                 before t_if_enter - Δpre
β_target(t) smoothly rises to 1                 during [t_if_enter - Δpre, t_if_enter]
β_target(t) ≈ 1                                 during [t_if_enter, t_if_exit]
β_target(t) smoothly decays to 0                during [t_if_exit, t_if_exit + Δpost]
β_target(t) = 0                                 after recovery
```

## 7.4 leakage control

这是建议中最值得保存到项目主线的细节之一。

为了抑制 stable 段 residual 外溢，建议显式加入：

```text
L_leak = |β_if(t) · ΔFθ(t)|     for stable / non-interface windows
```

或：

```text
L_stable_residual = |F_pred(t) - F_ref(e)|     for stable windows
```

这正对应我们当前已经观察到的核心问题：

- `delta_stronger` 可以把 residual 推大
- 但 stable 段会开始出现不必要修正

## 7.5 PhaseAwareStabilizer

建议在项目中明确加入这一概念。

功能包括：

- gate smoothing
- hysteresis
- minimum dwell time
- smooth activation/deactivation ramp

其位置建议为：

```text
β_raw(t) -> PhaseAwareStabilizer -> β_if(t)
```

## 7.6 SafetyFilter

安全过滤应保留，但作为工程部署层，不作为论文主贡献。

至少建议包括：

- force clipping
- force-rate limit
- conservative fallback around `F_ref`

## 8. 推荐实验路径

我认为工程师给出的实验路径是合理的，建议按以下顺序推进。

## Step 1：固定当前最强 `reference_anchor` 主线

起点建议使用当前最强的 `reference_anchor_delta_balanced`。

这一阶段的目标不是换系统，而是先把主线锚住。

## Step 2：审查 `reference_force`

需要输出：

- `reference_quality_report.csv`
- `reference_force_distribution.png`
- `bad_reference_cases.json`

重点检查：

- reference window 是否足够稳定
- `reference_force_std` 是否异常
- 同对象不同 episode 是否漂移过大
- fail 样本是否污染 `F_ref`
- `F_ref` 是否因果可获得

## Step 3：构造 `soft gate target`

先使用固定参数：

- `Δpre = 0.3s`
- `Δpost = 0.5s`

再做 sweep：

- `Δpre ∈ {0.2s, 0.3s, 0.5s}`
- `Δpost ∈ {0.3s, 0.5s, 0.8s}`

这一阶段的目的，是先把“界面前提前介入、界面后平滑退出”做成显式监督目标。

## Step 4：训练 `oracle gate delta`

训练形式：

```text
F_pred = F_ref + β_target · ΔFθ
```

这一步用于回答：

- residual 本身能不能学
- `ΔF_target` 是否合理
- `F_ref` 是否足以支撑最终力预测

这一阶段应尽量避免 learned gate 干扰 residual 学习。

## Step 5：训练 `learned gate`

训练形式：

```text
β_if = SoftGateHead(p_medium, context)
F_pred = F_ref + β_if · ΔFθ
```

重点是做对照：

- oracle gate vs learned gate

如果 learned gate 明显退化，说明问题在 gate，不在 delta。

## Step 6：加入 `PhaseAwareStabilizer`

加入：

- smoothing
- hysteresis
- minimum dwell time

观察：

- stable leakage 是否下降
- force jump 是否下降
- interface hit rate 是否保持

## Step 7：加入 `stable leakage loss`

这是建议中最值得优先实施的一步之一。

目标是得到：

- interface 段更积极
- stable 段不乱动

也就是解决当前“推大 delta 与 stable 外溢之间的冲突”。

## Step 8：做 `B-lite` 对照

训练形式：

```text
F_pred = F_ref + ε_stable + β_if · ΔF_interface
```

只做轻量双头，不做完整专家切换。

若 `B-lite` 没有明显超过主线，则不建议把专家切换式思路写进论文主方法。

## 12. 最终结论

我对这份《建议设计.docx》的最终判断是：

**整体合理，且值得采纳为当前项目的统一建议版设计。**

其中最应该保留的核心是：

1. 不再回到 absolute force regression
2. 不把 full expert switching 作为当前主方法
3. 以 `reference_force` 作为 episode-level 锚点
4. 以 `soft-gated residual delta` 作为核心控制形式
5. 把专家切换里的平滑机制、滞回、驻留时间、安全层吸收进主线
6. 用 `stable leakage loss` 和 `PhaseAwareStabilizer` 解决当前 residual 外溢问题

因此，项目的推荐主路线可概括为：

```text
reference_anchor_delta_balanced
-> soft gate target
-> oracle gate delta
-> learned gate
-> phase-aware stabilizer
-> stable leakage loss
-> optional B-lite ablation
```

一句话总结：

**这份建议最成功的地方，不是把系统做得更复杂，而是把当前最有证据支持的方法主线重新组织得更清楚、更可实现、也更适合论文表达。**
