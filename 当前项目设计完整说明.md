## 2. 项目目标与问题定义

本项目研究的是“跨介质视触抓取”。

任务背景是：

- 机械臂在水下抓取目标物体
- 随后抓着物体上提
- 过程中穿过水-气界面
- 最终在出水后继续维持抓取稳定

与普通抓取相比，这里的特殊困难在于：

1. 介质状态变化明显  
   水下、界面中、出水后，对抓取力需求不同。

2. 视觉和触觉都重要  
   单纯看视觉无法稳定判断抓取状态，单纯看触觉也难以知道介质变化的空间上下文。

3. 控制目标不是一个静态值  
   稳态段、界面段、恢复段的合理力输出不是同一种模式。

因此项目最初试图训练一个统一策略模块，让模型基于视觉、触觉和介质状态，直接回归合理抓取力。

随着实验推进，项目对控制问题的理解逐渐从：

```text
直接从多模态输入预测最终抓取力
```

演化为：

```text
最终控制力 = 基准力 + 界面修正量
```

也就是更结构化的形式：

```text
force_pred = base_force + interface_gate * interface_delta
```

进一步地，当前最被验证有效的主线已经变成：

```text
force_pred = reference_force + gate * delta
```

后续再考虑扩展到：

```text
force_pred = reference_force + base_correction + gate * delta
```

这意味着：  
当前项目不再把“绝对力从零开始盲猜”视为唯一正确的问题定义，而是越来越强调**锚点 + 修正量**的控制形式。

## 3. 当前数据与标签体系

### 3.1 数据规模

当前处理后数据大致规模为：

- 样本数：287
- 窗口数：24,611
- Water 窗口：9,909
- Interface 窗口：3,586
- Air 窗口：11,116
- stable 样本：262
- fail 样本：25

对当前项目来说，这个量级足以支持：

- 阶段化训练
- 小规模对照实验
- 局部过拟合 sanity check
- `fold1` 级别验证

但不足以支撑大量无约束的高复杂度策略范式探索，因此项目训练必须尽量建立在清晰的结构假设上。

### 3.2 样本内容

每个样本通常包括：

- 视频
- 触觉时序
- 标注文件

标注中包括与流程阶段相关的关键时间点，例如：

- `t_grasp_stable`
- `t_if_enter`
- `t_if_exit`

这些时间点在项目中有两个用途：

1. 划分介质阶段
2. 构造控制监督目标

### 3.3 当前标签语义

项目里最重要的标签，不只是“当前介质是什么”，还包括：

1. **phase / medium labels**
   - Water
   - Interface
   - Air

2. **reference supervision**
   - 哪些窗口可以被视为稳定参考段

3. **delta supervision**
   - 哪些窗口属于界面修正段

4. **control force targets**
   - 当前窗口控制意义上的目标力

5. **reference_force**
   - 当前 episode 内稳定参考力

6. **delta_force_target**
   - 相对于 `reference_force` 的修正量

### 3.4 当前关于 `reference_force` 的关键认识

项目早期默认倾向于把 `reference_force` 当作相对稳定、可学习的“对象级基准力”。

但后续 `fold1` 分组统计已经说明：

- `reference_force` 在对象内也会明显漂移
- 在相同条件组内也会漂移很大

因此当前项目已经不再把 `reference_force` 理解成“稳定对象常数”，而是更接近：

**依赖当前 episode 执行状态的样本级锚点。**

这条认识直接改变了当前策略主线。

## 5. 当前训练链路是怎么组织的

项目最初按阶段拆成：

1. `Stage1` 感知
2. `Stage2` 视触对齐
3. `Stage3 / 36` 策略头训练与诊断
4. `Stage4 / 4b` 联合训练或策略微调

### 5.1 Stage1：感知

目标：

- 让模型具备基础介质识别能力
- 让视觉/触觉特征能够支持后续控制问题

结论：

- 当前感知已经可用
- 特别是 `Interface` 的识别能力明显优于早期版本

### 5.2 Stage2：对齐

目标：

- 让视触表示在对象和过程层面更一致

结论：

- 工程链路打通
- 但对最终控制收益不是稳定、直接的

### 5.3 Stage3 / 36：策略头诊断

目标：

- 更细粒度地观察策略头能不能学到合理力输出

这一层最重要的价值，不是产出一个最终最强模型，而是帮助项目逐步确认：

- `base` 能不能学
- `delta` 能不能学
- gate 是否是主要瓶颈
- 多项 loss 是否彼此冲突

### 5.4 Stage4 / 4b：联合训练或联合微调

目标：

- 在完整系统下得到最终可用控制表现

结论：

- 感知已经够用
- 但控制侧仍然是主要短板

## 6. 当前项目是怎样一步步收敛到“reference anchor 主线”的

这部分对工程师最重要，因为它解释了：  
为什么现在项目主线不是最早设想的那种“统一 full joint 回归”。

### 6.1 第一阶段认识：`delta` 并不是学不会

通过 OBJ012 tiny 过拟合实验，已经明确看到：

- 如果 `base` 不稳定，`delta` 很容易塌到 0
- 只要把 `base` 换成 `oracle/reference`，`delta` 立刻能学起来

所以项目第一次收敛出的关键认识是：

**问题不在 residual 分支本身，而在锚点是否稳定。**

### 6.2 第二阶段认识：`base` 在小集上能学，但对尺度很敏感

继续做 tiny 实验后又确认：

- pure `base` 不是完全学不会
- 但它对数值尺度极其敏感

通过 `base_normalization_scale` 之后，tiny 上的 `base/reference` 子问题能学顺。

项目第二次收敛出的关键认识是：

**base 的问题既有优化尺度问题，也有目标本身的问题。**

### 6.3 第三阶段认识：joint 不是完全没用，但不是当前主武器

后续的 `learned base frozen + delta_norm` 和 `light joint` 试验说明：

- 拆开学是通的
- 再轻量 joint，可以在 train 上把两者接起来

但 val 上真正的泛化短板仍主要在 `base`。

所以项目第三次收敛出的认识是：

**joint 不是当前最先要解决的问题，锚点定义才是。**

### 6.4 第四阶段认识：`reference_force` 本身就是样本级锚点

`fold1` 的 `reference_force` 分组统计最终把问题彻底说清：

- `reference_force` 不是稳定对象常数
- 它更像当前 episode 的执行级参考量

于是项目主线从：

```text
learned absolute base + delta
```

转到：

```text
measured reference_force + delta
```

这是当前项目最重要的结构性转向。

## 7. 当前策略主线的正式定义

当前最被验证有效的主线可以概括成：

### 7.1 锚点定义

- 不再让模型纯学习 absolute `base`
- 直接使用 measured `reference_force` 作为当前 episode 的基准锚点

### 7.2 策略学习目标

- 模型主要学习 `delta`
- 也就是在锚点基础上该补多少

### 7.3 主线形式

当前最被验证有效的表达是：

```text
force_pred = reference_force + gate * delta
```

这里：

- `reference_force` 负责大基准
- `delta` 负责界面段补偿
- gate 负责是否放大/抑制修正量

### 7.4 当前工程含义

这意味着项目当前主线更像一个：

**reference-anchored residual control model**

而不是一个：

**from-scratch absolute force regressor**

## 8. 当前已经验证过的几条关键策略配置

这一部分是为了让工程师知道：  
哪些实验只是试过，哪些已经成为设计决策依据。

### 8.1 `stage36b_policy_fold1_base_norm`

用途：

- 作为 pure learned absolute base 的正式 baseline

结论：

- 基本失败
- `base` 在 `fold1` 上没有真正学到正确量级

意义：

- 它保留为对照组
- 不再视为主线

### 8.2 `stage36b_policy_fold1_reference_anchor_delta_norm`

用途：

- 使用 `reference_force` 直锚
- 只训练 `delta`

结论：

- 相比 `fold1_base_norm`，控制指标有数量级改善
- `base_minus_reference_mean = 0`

意义：

- 这是当前主线成立的核心证据

### 8.3 `stage36b_policy_fold1_reference_anchor_delta_stronger`

用途：

- 在直锚基础上更强地推动 `delta`

结论：

- `delta` 的幅值确实被往前推了一步
- 但开始出现 stable 段外溢
- 最佳点出得很早，后面容易过推

意义：

- 说明当前剩余问题确实是 `delta` 学得不够充分
- 也说明不能一味更猛

### 8.4 `stage36b_policy_fold1_reference_anchor_delta_balanced`

用途：

- 在 `norm` 和 `stronger` 之间找一个更平衡的版本

设计思路：

- 保留更强的 interface 推力
- 同时加回 stable 段约束
- 减少过推

意义：

- 当前已经从“找主线”进入“调手感”阶段

## 11. 当前项目的主要局限

### 11.1 `delta` 还没有完全学透

虽然当前主线已经正确，但界面段的 residual 仍偏保守，尤其在复杂样本上补偿还不够充分。

### 11.2 stable/interface 平衡仍在调

一旦把 `delta` 推猛，就容易出现 stable 段外溢。  
这说明系统还在寻找“足够积极但不过推”的平衡点。

### 11.3 `reference_force` 的标签稳定性仍值得继续审查

虽然当前已经把它当锚点使用，但：

- reference window 的选取方式
- 统计方式
- 极端样本质量

仍然是影响上限的重要因素。
