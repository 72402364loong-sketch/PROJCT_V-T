# 界面增强独立测试版设计草案

更新日期：2026-05-01

## 1. 目标

在**不修改当前主线逻辑**的前提下，新增一套“界面增强”独立测试版，用于验证下面这个想法是否有效：

> 根据标签读取 `t_grasp_stable -> t_if_exit` 这段时间内的视觉与触觉数据，把它作为“界面增强上下文”，专门提升模型在界面段的控制表现。

这里的核心目标不是替代当前主线，而是作为一个**旁路实验版本**，回答：

1. 界面前后的连续时序上下文，能否帮助模型更好地学 `delta`
2. 界面段能力提升后，是否会明显带来 stable 段外溢
3. 这条路值不值得再合并回主线

## 2. 设计原则

本测试版必须满足以下约束：

1. **不改当前主线的数据逻辑**
   - 不改现有 `dataset.py`
   - 不改现有主训练 stage 配置
   - 不影响当前 `reference_anchor` 主线

2. **只新增，不覆盖**
   - 新增脚本
   - 新增实验数据索引
   - 新增实验 dataset / collator / config
   - 新增独立训练入口或独立 stage

3. **先做最小可验证版本**
   - 先验证“界面上下文是否有用”
   - 暂不追求和主线完全融合

## 3. 关键思想

这个想法不能简单理解成：

> 把 `t_grasp_stable -> t_if_exit` 整段都直接标成 interface 数据

这样做风险很大，因为会把本来应该是“稳定参考段”的窗口污染成界面段。

更合理的理解应该是：

### 3.1 把整段分成两个角色

1. **参考上下文段**
   - 时间范围：`[t_grasp_stable, t_if_enter)`
   - 作用：提供界面前稳定状态的视觉/触觉上下文
   - 语义：告诉模型“进入界面之前正常状态长什么样”

2. **界面目标段**
   - 时间范围：`[t_if_enter, t_if_exit]`
   - 作用：提供真正需要增强学习的界面窗口
   - 语义：告诉模型“进入界面后应该怎么改力”

所以这个测试版的核心不是“重标整段”，而是：

**让 `t_grasp_stable -> t_if_exit` 成为一个有内部结构的界面增强样本。**

## 4. 推荐的最小可落地版本

### 4.1 数据单位

主线当前的数据单位是 window。  
这个独立测试版建议把一个训练样本定义成：

- 一小段 **reference context windows**
- 一小段 **interface target windows**

也就是说，一个 sample 不再只是单个窗口，而是一个“界面增强片段”。

### 4.2 每个增强样本包含什么

建议每个增强样本至少包含：

- `sample_id`
- `object_id`
- `t_grasp_stable`
- `t_if_enter`
- `t_if_exit`
- `context_window_indices`
- `interface_window_indices`
- `reference_force`
- `delta_force_targets`
- `control_force_targets`

其中：

- `context_window_indices` 来自 `[t_grasp_stable, t_if_enter)`
- `interface_window_indices` 来自 `[t_if_enter, t_if_exit]`

### 4.3 第一版不需要重做原始预处理

第一版建议**不重新生成视频/触觉特征文件**，只新增一个索引构建脚本：

- 读取现有样本标签
- 读取现有 processed windows
- 根据时间戳把窗口切成：
  - context windows
  - interface windows
- 最终导出一个 manifest

## 5. 新增文件建议

以下文件全部为**新增文件**，不改现有逻辑。

### 5.1 数据索引构建脚本

建议新增：

`scripts/build_interface_context_manifest.py`

作用：

- 遍历当前 split 中的样本
- 找出每个样本里：
  - `t_grasp_stable`
  - `t_if_enter`
  - `t_if_exit`
- 将现有窗口按时间划分成：
  - context windows
  - interface windows
- 过滤掉：
  - 没有 `t_grasp_stable`
  - 没有 `t_if_enter`
  - 没有 `t_if_exit`
  - 没有足够 context 或 interface windows
  的样本

输出建议：

`data/processed/interface_context/fold1_manifest.csv`

每行一个“界面增强片段样本”。

### 5.2 独立 dataset

建议新增：

`src/cmg/data/interface_context_dataset.py`

作用：

- 读取 `fold1_manifest.csv`
- 内部复用现有 processed windows / 视觉 / 触觉读取逻辑
- 返回一个结构化样本：
  - `context_visual`
  - `context_tactile`
  - `target_visual`
  - `target_tactile`
  - `reference_force`
  - `delta_force_targets`
  - `control_force_targets`
  - `context_mask`
  - `target_mask`

### 5.3 独立 collator

建议新增：

`src/cmg/data/interface_context_collator.py`

作用：

- 处理可变长度的 context windows / interface windows

- 生成：

### 5.4 独立 stage 配置

建议新增：

`configs/stages/stage36b_policy_fold1_interface_context_reference_anchor.yaml`

作用：

- 只给测试版使用
- 不干扰当前 `reference_anchor_delta_*` 主线

## 6. 第一版模型/训练建议

为了保证是“独立测试版”，第一版不要改主线模型文件里的现有逻辑，而是新增一个实验性轻量版本。

## 6.1 第一版最小模型目标

不追求新结构很复杂，只做一件事：

**让 `delta` head 在预测当前 interface 窗口时，额外看到一份来自 `t_grasp_stable -> t_if_enter` 的上下文摘要。**

### 6.2 建议的最小实现

做法：

1. 先用现有视觉编码器、触觉编码器编码每个 context window
2. 对 context windows 做简单池化：
   - mean pooling
   - 或 attention pooling
3. 得到一个 `interface_context_embedding`
4. 在预测 interface target windows 的 `delta` 时，将这个 embedding 拼到 residual head 输入里

公式上可理解为：

```text
delta_pred = f(current_window_features, interface_context_embedding)
force_pred = reference_force + gate * delta_pred
```

### 6.3 为什么第一版不建议更复杂

第一版不建议一开始就做：

- cross-attention 大改结构
- 新 backbone
- 复杂时序 transformer

原因是：

- 当前最需要回答的是“界面上下文有没有用”
- 不是“最复杂结构能不能再涨一点”

先用最朴素版本把因果关系看清最值钱。

## 7. 第一版训练目标建议

第一版建议坚持当前已经被验证过的主线思想：

- `base` 继续用 `reference_force` 直锚
- 不再让模型纯猜 absolute base
- 重点只增强 `delta`

### 7.1 监督设计

建议：

1. **context 段**
   - 不直接当 interface 目标
   - 只作为上下文编码输入

2. **interface 段**
   - 继续使用现有 `delta_force_target`
   - 继续使用现有 `control_force_target`

### 7.2 loss 设计

第一版建议尽量贴当前 `reference_anchor` 主线：

- `residual_interface_weight`
- `residual_stable_zero_weight`
- `residual_non_interface_weight`
- `delta_normalization_scale`

也就是说，先不要发明新 loss，只是让当前 residual loss 在“更有上下文”的输入上训练。

## 8. 实验版本划分建议

建议把独立测试版分成三层，不要一步到位。

### V0：只做数据旁路，不改模型

做法：

- 只生成 `interface_context_manifest`
- 只对样本做“界面增强采样”
- 仍然喂给当前 `reference_anchor` 模型

它回答：

**如果只提高界面样本覆盖率，不引入上下文摘要，能涨多少？**

### V1：加 context embedding

做法：

- 使用 context windows 编码得到 `context_embedding`
- 拼到 residual head 输入

它回答：

**真正的“界面上下文”是否带来额外收益？**

### V2：再考虑更复杂时序建模

只有在 V1 明显有效后，才考虑：

- temporal attention
- context-target cross attention
- 细粒度动态对齐

## 9. 与当前主线的关系

这个方案必须明确定位为：

**旁路实验版，而不是立即替代主线。**

当前主线仍然应该是：

- `reference_anchor_delta_norm`
- `reference_anchor_delta_stronger`
- `reference_anchor_delta_balanced`

这个新测试版的意义是补充回答：

> 如果在 `delta` 学习中显式引入“界面前稳定上下文”，能不能进一步提升界面段表现？

## 10. 评估重点

这个独立测试版不应该只看一个总指标。

重点应固定看：

1. `control_interface_mae`
2. `control_interface_hit_rate_200`
3. `control_interface_bias`
4. `delta_minus_target`
5. `stable_expert` 上的 `force_interface_delta`
6. `stable_expert` 上的 `gated_residual`

判断标准：

- 如果 `interface` 指标继续变好
- 且 stable 段外溢没有明显变坏

则说明这条思路值得继续投入。

如果：

- `interface` 只涨很少
- 但 stable 段明显被污染

则说明“整段上下文增强”需要更克制的使用方式。

## 11. 风险点

### 11.1 最大风险

把 `t_grasp_stable -> t_if_enter` 误当成 interface 目标来学。

这样会导致：

- `delta` 在 stable 段也开始乱出手
- 模型整体变得更激进

### 11.2 第二个风险

界面片段增强后，训练分布和真实推理分布差太远。

所以测试版里必须保证：

- 训练时仍保留当前窗口级控制任务
- 只是多给一份界面上下文

不能完全把任务改造成另一种问题。

## 12. 推荐的最小实施顺序

### 第一步

只新增：

- `build_interface_context_manifest.py`
- `fold1_manifest.csv`

先确认：

- 有多少样本带完整 `t_grasp_stable -> t_if_exit`
- 每个样本平均有多少 context windows
- 每个样本平均有多少 interface windows

### 第二步

做 V0：

- 只改采样
- 不改模型

### 第三步

做 V1：

- 加 `context_embedding`
- 只增强 `delta`

### 第四步

只有在 V1 明显有效后，再考虑并回主线。

## 13. 一句话总结

这个想法是值得做的，但最好的落地方式不是“把 `t_grasp_stable -> t_if_exit` 整段重标成 interface”，而是：

**把这整段做成一个带内部结构的界面增强片段，其中 `t_grasp_stable -> t_if_enter` 提供参考上下文，`t_if_enter -> t_if_exit` 提供界面目标监督，并以完全独立的测试版形式实现。**
